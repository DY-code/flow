# 基于 GitHub Actions 的任务计划未排期邮件提醒方案

## 1. 项目概述

### 1.1 基本定位
为“任务计划”提供一套独立于本地 Flow 项目的自动提醒机制。系统部署在 GitHub Actions 上，定期检查仓库中的“任务计划”内容，识别长期未设定执行时间的任务，并通过邮件提醒用户及时补充计划。

### 1.2 功能概述
系统每周自动运行一次，从保存“任务计划”的 GitHub 仓库中读取数据，对符合条件的任务进行初步筛选、脱敏和裁剪，再调用语言模型进行优先级判断与邮件撰写，最后通过 QQ 邮箱 SMTP 发送汇总提醒邮件。

### 1.3 核心特性
- 与本地 Flow 项目使用隔离，不依赖本地运行环境
- 使用 GitHub Actions 作为免费云端自动执行环境
- 仅对“未完成 + 叶子任务 + 未填写日期”的任务进行提醒
- 支持基于语言模型从候选任务中筛选最需要提醒的 3 项
- 邮件开头包含该周任务完成情况
- 支持连续多周重复提醒，直到任务补上日期或完成
- 对敏感内容先脱敏和裁剪，再发送给模型
- 使用 Actions 日志和 artifact 保存运行结果，便于排查

## 2. 部署与运行方式

### 2.1 部署方式
采用“同仓库 GitHub Actions 定时工作流”方案。

即：
- “任务计划”所在 GitHub 仓库中同时保存：
  - Flow 项目文件
  - GitHub Actions 工作流文件
  - 执行提醒逻辑的脚本文件

### 2.2 触发方式
仅使用 GitHub Actions 的 `schedule` 定时触发。

说明：
- 支持使用 cron 表达式自定义触发时间
- 不启用手动触发
- 不依赖 push / pull_request 触发

### 2.3 仓库读写边界
- GitHub Actions 可以读取仓库中的“任务计划”文件
- 工作流运行时可在临时环境中生成和处理文件
- 默认不回写仓库内容，不自动 commit / push 任何结果
- 后续 Flow 项目继续推送到该仓库时，不会主动影响 `.github/workflows` 下的工作流文件

## 3. 任务筛选规则

### 3.1 数据来源
数据来源为“任务计划”项目文件。

### 3.2 硬性前置条件
程序先根据以下硬条件筛选候选任务：
- 必须是叶子任务
- 必须是未完成任务
- 任务标题末尾的 `[]` 中没有日期字段

### 3.3 长期未排期判定规则
对满足硬条件的任务，再按“最后修改时间”判断是否进入提醒候选范围：

- 若最近 7 天内未补充日期
- 且最近 7 天内无内容调整

则认为该任务属于“长时间未指定执行时间”的候选任务。

### 3.4 重复提醒规则
- 同一任务允许连续多周重复提醒
- 一旦任务补上日期，立即从后续提醒中移除
- 一旦任务完成，也从后续提醒中移除

## 4. 模型参与策略

### 4.1 模型使用目标
语言模型用于：
- 根据候选任务摘要判断最值得提醒的任务
- 组织和撰写邮件正文
- 在邮件开头生成该周任务完成情况总结

### 4.2 模型边界
程序不将整个“任务计划”全文直接交给模型，而是先完成初筛，再将候选任务摘要发送给模型。

模型不负责突破硬性筛选条件，仅在候选集合内完成优先级判断与文案组织。

### 4.3 数据安全策略
考虑到“任务计划”中可能包含敏感内容，程序在调用模型前需要：
- 对候选任务内容进行脱敏
- 对候选任务信息进行裁剪
- 仅保留模型完成判断和写信所需的最小必要摘要信息

### 4.4 模型调用方案
采用：
- OpenRouter
- 单轮结构化提示调用

输入内容包含：
- 该周任务完成情况所需摘要
- 候选任务摘要列表

输出目标包含：
- 该周任务完成情况总结
- 最终选中的最多 3 项任务
- 对应提醒理由
- 最终邮件正文

## 5. 邮件提醒策略

### 5.1 邮件类型
每周发送 1 封汇总邮件。

### 5.2 提醒项数量限制
- 若候选任务不超过 3 项，则全部进入提醒邮件
- 若候选任务超过 3 项，则由模型从中筛选最合适的 3 项进行提醒

### 5.3 邮件内容结构
建议邮件正文由以下部分组成：
1. 本周任务完成情况
2. 本周最值得尽快制定计划的任务
3. 每项任务的提醒理由
4. 温和的行动建议或结尾提醒

其中：
- “本周任务完成情况”通过提示词约束模型在邮件开头生成
- 提醒文案风格应偏简洁、明确、可执行

## 6. 邮件发送方案

### 6.1 发件与收件配置
- 固定 QQ 邮箱作为发件邮箱
- 固定 QQ 邮箱作为收件邮箱

### 6.2 发送协议
使用 QQ 邮箱 SMTP 服务发送邮件。

### 6.3 失败重试策略
采用“QQ SMTP + 程序内重试”方案：
- 首次发送失败后重试 2 次
- 若最终仍失败，则本次工作流标记为失败

## 7. 密钥与配置管理

### 7.1 配置存放方式
所有敏感配置均保存在 GitHub Secrets 中，不放入仓库文件。

### 7.2 典型配置项
包括但不限于：
- GitHub 访问所需配置
- OpenRouter API Key
- QQ SMTP 授权码
- 发件邮箱地址
- 收件邮箱地址

### 7.3 安全原则
- 仅在工作流运行时注入环境变量
- 不在日志中输出敏感信息
- 不将完整敏感任务内容直接发给模型

## 8. 结果记录与可观测性

### 8.1 运行记录方式
使用：
- GitHub Actions 日志
- GitHub Actions artifact

### 8.2 建议记录内容
每次运行至少记录：
- 是否成功读取任务计划
- 初筛出的候选任务数量
- 发送给模型的摘要内容
- 模型返回结果
- 最终进入提醒邮件的任务
- 邮件发送是否成功

### 8.3 失败处理
- 邮件发送失败时，GitHub Actions 工作流直接标红
- 模型输入摘要和模型输出结果保存为 artifact，便于后续排查

## 9. 建议的处理流程

整体处理流程如下：

1. GitHub Actions 根据 `schedule` 定时触发
2. 检出仓库代码
3. 读取“任务计划”文件
4. 程序按硬条件筛选候选任务
5. 程序按最后修改时间判断是否属于长期未排期任务
6. 程序对候选任务进行脱敏与裁剪
7. 程序整理“本周任务完成情况摘要 + 候选任务摘要列表”
8. 调用 OpenRouter
9. 模型返回任务筛选结果与邮件内容
10. 程序通过 QQ SMTP 发送汇总邮件
11. 将运行信息写入 Actions 日志，并保存 artifact
12. 若邮件发送失败，则工作流标红

## 10. 注意事项

- GitHub Actions 绑定在具体仓库上，因此工作流文件需要放入“任务计划”所在仓库
- 当前 Flow 的推送逻辑主要提交项目文件，不会主动覆盖工作流文件
- `schedule` 支持自定义时间，但后续实现时需要明确时区
- 如果未来希望增强稳定性，可进一步约束模型输出格式，减少自动化运行中的不确定性
- 如果后续需要历史统计、人工复核、网页查看记录等能力，再考虑扩展为更完整的服务架构
